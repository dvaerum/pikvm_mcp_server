# Autonomous roadmap — pointer-prediction pipeline

Living document. Each cron fire reads this, picks the highest-priority **`[ ]` pending** step, executes it end-to-end (including live verification), and updates this file with results before returning. Mark steps `[x]` once verified, `[~]` if partially done with a note, `[!]` if blocked with a description of the blocker.

**North-star goal:** ≥90 % real-launch click rate on icon-sized iPad targets *without* slam-fallback, by combining (a) a cursor-position detector that's accurate enough to skip detection-side fallbacks and (b) a forward pointer-acceleration model that lets `moveToPixel` predict where the cursor will land from an emit sequence.

## Cross-cutting rules

- **No verdicts from samples.** If a bench shows 0 %, the next action is the next experiment, not a "root cause" doc from 3 of 27 frames. Sample all data or label conclusions as hypotheses (see `feedback_no_verdicts_from_samples` memory).
- **Screenshots are source of truth.** Always visually inspect intermediate frames before concluding a model is working (`feedback_screenshots_source_of_truth`).
- **Detector residual is not ground truth.** Don't infer "iPad ignored the tap" from detector-reported residuals (`feedback_detector_residual_is_not_ground_truth`).
- **MPS-CPU tensor co-location.** All PyTorch arithmetic ops on MPS must have operands on the same device, or values silently come back wrong (`feedback_mps_cpu_tensor_subtraction_silent_bug`).
- **REJECTED CLAIMS list** in `feedback_rejected_unverified_claims` — don't assert "pointer-effect snap" / "rate-limit dead zone" / "iPad ignores tap" without citing a screenshot, code path, or this-session data.

## Cron-fire workflow (read this first when fired)

1. `git pull --rebase --autostash` if there's a remote — there isn't one yet, so just survey current branch.
2. `git status` — clean tree or honest stash before starting.
3. Read this whole doc. Pick the highest-priority unchecked step.
4. Execute it. Live-test where applicable. Save screenshots when human eyes might disagree with metrics.
5. Update this doc: mark step done, record actual results inline.
6. `git add` the work; commit with a message that names the step (e.g. `feat(roadmap): 1.5 wire pointer-accel into move-to`).
7. Stop. Don't try to do the next step in the same fire — let the next cron handle it. (Exception: trivial follow-ups like updating the plan doc.)
