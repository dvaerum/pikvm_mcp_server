# Autonomous roadmap — pointer-prediction pipeline

Living document. Each cron fire reads this, picks the highest-priority **`[ ]` pending** step, executes it end-to-end (including live verification), and updates this file with results before returning. Mark steps `[x]` once verified, `[~]` if partially done with a note, `[!]` if blocked with a description of the blocker.

**North-star goal:** ≥90 % real-launch click rate on icon-sized iPad targets *without* slam-fallback, by combining (a) a cursor-position detector that's accurate enough to skip detection-side fallbacks and (b) a forward pointer-acceleration model that lets `moveToPixel` predict where the cursor will land from an emit sequence.

- [!] **1.12 Open-loop move-to prototype — Phase 1 ships, Phase 2 fails compounding-error gate** (2026-06-02):

  Implementation:
  - `src/pikvm/open-loop-planner.ts` (new) — greedy chunk-by-chunk
    planner, cardinal-only, MAX_EMITS=50 termination guard, returns
    `{emits, residualPx, hitMaxEmits, predictorFailed}`.
  - `src/pikvm/__tests__/open-loop-planner.test.ts` (new) — 9 unit
    tests using stub forward models. All pass. Confirms planner
    algorithm correctness (convergence, sign handling, axis dominance,
    feature plumbing through `buildFeatures`, hitMaxEmits + predictor-
    null reporting).
  - `benches/bench-1.12-phase2-replay.ts` (new) — Phase 2 replay
    against 40 target displacements (5 magnitudes × 8 directions,
    100-800 px each axis).
  - `benches/bench-1.12-phase2-debug-signs.ts` (new) — single-emit
    diagnostic that proved v2-wider's signs ARE correct at all
    dt_prev values; the Phase 2 failure isn't a sign-flip bug.

  Phase 2 result: median Euclidean residual **721 px** vs target gate
  **<10 px**. 23/40 cases hit MAX_EMITS=50 without converging.
  Pattern: small targets (+x_100, +x_200, -y_100, etc.) converge in
  <20 emits; large targets (anything with +y, anything ≥400 px)
  diverge.

  Root cause (investigated, not papered over): **multi-step error
  compounding**. v2-wider was trained for single-step 50 ms horizon
  prediction with per-step MAE ~3 px on burst, ~10 px on chunkedBurst-y
  (offline numbers from 1.10). When the planner queries it sequentially
  for 50 chunks, the simulated cursor state drifts further from the
  trained distribution at each step, and the per-step errors grow
  non-linearly. The sign-correct single-emit predictions (verified via
  debug-signs probe) do not imply correctness when chained into a
  closed loop.

  Mechanism that hides this in production: live `moveToPixel` takes
  one `predictDisplacement` call per move and uses `Math.abs()` on the
  output (`learnedBallisticsPxPerMickey` at `move-to.ts:153-154`) —
  the prediction is used only as a magnitude estimate, and the sign
  comes from the command vector. So v2-wider's 1.11 live win is real
  but doesn't transfer to the multi-step open-loop role 1.12 wants.

  Implication for the roadmap: 1.12 as designed needs a different
  forward model. Options for unblocking (none cheap; all real work):
  - (i) Train a model with a longer prediction horizon (e.g., predict
    cursor position after a 16-emit chunked-burst, not after 50 ms).
    Trajectory data from 1.9 already has the inputs but would require
    rewriting `build_examples` in the trainer.
  - (ii) Sequence model (RNN/transformer) that propagates internal
    state, so multi-step compounding learns to self-correct.
  - (iii) Drop open-loop ambition; use v2-wider with the existing
    closed-loop correction (1.11 ships 80% HIT — possibly sufficient
    for moveToPixel's "good enough" target).

  **Status: 1.12 paused, marked blocked.** Phase 3 (live iPad test)
  was scheduled to compound the same errors against real cursor
  dynamics — running it would not produce a usable result. Phase 4
  (combined system A/B) is gated on Phase 3. The planner module +
  test infrastructure remain in tree for whichever forward-model
  redesign (i)/(ii) wins.

## Cross-cutting rules

- **No verdicts from samples.** If a bench shows 0 %, the next action is the next experiment, not a "root cause" doc from 3 of 27 frames. Sample all data or label conclusions as hypotheses (see `feedback_no_verdicts_from_samples` memory).
- **Screenshots are source of truth.** Always visually inspect intermediate frames before concluding a model is working (`feedback_screenshots_source_of_truth`).
- **Detector residual is not ground truth.** Don't infer "iPad ignored the tap" from detector-reported residuals (`feedback_detector_residual_is_not_ground_truth`).
- **MPS-CPU tensor co-location.** All PyTorch arithmetic ops on MPS must have operands on the same device, or values silently come back wrong (`feedback_mps_cpu_tensor_subtraction_silent_bug`).
- **REJECTED CLAIMS list** in `feedback_rejected_unverified_claims` — don't assert "pointer-effect snap" / "rate-limit dead zone" / "iPad ignores tap" without citing a screenshot, code path, or this-session data.
- **Pair every live A/B with an iPadCollector ground-truth bench.** The live production bench's residuals come from the screenshot detector and can lie. iPadCollector (`ipad-collector/`, port 8767) exposes ground-truth cursor positions and ground-truth tap locations via WS. `showScene({kind: 'image', ...})` accepts arbitrary images — including a PiKVM screenshot of the real home screen — so you get the same visual surface as production PLUS ground truth. Example: `benches/bench-1.13c-groundtruth-pointer-accel.ts`, `benches/bench-5.1-groundtruth-retry-loop.ts`. Skipping this is how the 1.11 "+55pp pointer-accel lift" claim survived 6 fires before 1.13c/1.14 walked it back. Full rationale: `feedback_ipadcollector_ground_truth` memory.

## Cron-fire workflow (read this first when fired)

1. `git pull --rebase --autostash` if there's a remote — there isn't one yet, so just survey current branch.
2. `git status` — clean tree or honest stash before starting.
3. Read this whole doc. Pick the highest-priority unchecked step.
4. Execute it. Live-test where applicable. Save screenshots when human eyes might disagree with metrics.
5. Update this doc: mark step done, record actual results inline.
6. `git add` the work; commit with a message that names the step (e.g. `feat(roadmap): 1.5 wire pointer-accel into move-to`).
7. Stop. Don't try to do the next step in the same fire — let the next cron handle it. (Exception: trivial follow-ups like updating the plan doc.)

If you finish all of Stages 1–4: cancel your own cron with `CronDelete` (the job ID from when you ran `CronList`) and write a final summary in this doc.
