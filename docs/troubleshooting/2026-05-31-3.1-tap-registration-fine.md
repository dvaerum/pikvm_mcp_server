# Step 3.1 — tap registration works; today's "NO_LAUNCH" was a coordinate bug

**Bench:** `benches/bench-tap-isolation.ts` (new, this step)
**Date:** 2026-05-31
**Result:** 7/10 HIT against a confirmed visible icon. Stage 3 (tap-registration) is **not the bottleneck**; today's earlier "85 % NO_LAUNCH" was the bench's hardcoded page-1 icon coords being run against an iPad that was sitting on **home page 2**, where those pixels are empty space.

## What happened

The production bench (`bench-click-production.ts`) hardcodes four targets — Settings (1027, 837), Books (757, 837), AppStore (1027, 702), Files (1162, 435) — assuming the iPad is on home page 1. Today's iPad was on **home page 2** (different icons: Clock, Weather, Stocks, …, Tips, SumUp Payment, Safari, iPadCollector). `ipadGoHome(forceHomeViaSwipe: true)` returns the iPad to the home screen the user last opened, not specifically page 1.

The bench was therefore clicking empty wallpaper:
- (1027, 837) is between SumUp Payment's column and Safari's column, below row 5 — empty pixel on page 2.
- (757, 837) is between Tips and SumUp Payment, also empty.
- (1162, 435) is empty.
- (1027, 702) accidentally lands very close to iPadCollector at (1067, 700) on page 2 — that's why AppStore "HIT 5/5" in the morning baseline. The bench's `verifyClickByDiff` only checks "screen changed", not "the right app launched". Those 5 "hits" were all iPadCollector launches.

The 85 % NO_LAUNCH rate was correct *as reported* — those clicks really didn't launch anything, because there was nothing under the cursor.

10 trials, all targeting **iPadCollector at HDMI (1067, 700)** — a real icon visible on the current iPad page. Bare `moveToPixel` + raw `client.mouseClick({ downMs: 150 })`, no retry / verification layer. Own pixel-similarity classifier vs a fresh home reference (avoids inheriting bias from `verifyClickByDiff`).

The three NO_LAUNCH trials all had **residual ≥ 90 px** — i.e. the cursor missed the icon. The seven HIT trials had **residual ≤ 31 px**. There's no case in the data where the cursor was on the icon AND the tap was rejected.

Post-click frame for trial 5 (HIT) was inspected manually: black screen with a Home Indicator bar — that's iPadCollector's launch view. The tap really did fire and the app really did open.

## Verdict — does iPadOS register taps when the cursor is perfectly placed?

**Yes.** When the cursor lands inside an icon's hit-area, iPadOS reliably launches the app. The "tap-registration is broken" hypothesis from this morning was wrong — it was a measurement artifact from the bench targeting the wrong screen.

## What this invalidates

- **PA37 "v11 → 5 %" decay narrative.** That bench measured against an iPad in an unknown home-page state. The 5 % HIT rate is consistent with "few coords accidentally land on icons on page 2".
- **Today's 1.6 baseline 15 %.** Same bench, same coord set, same wrong screen. The 3 HITs were iPadCollector launches via the AppStore-named target coincidentally landing near it.
- **Today's 1.6 treatment 5 %.** Same problem plus the (real) learned-ballistics bug that ballooned residuals.
- **Stage 3 mostly disappears.** 3.2 (click-duration sweep) and 3.3 (inter-click-delay sweep) were aimed at "iPadOS rejects taps at default duration/delay". The data doesn't show that pattern. 3.4 (snap-zone offset) is more nuanced — borderline-residual taps DO launch (trial 6 with residual 89 px still HIT), so the snap-zone is *forgiving*, not *brittle*.

## What's the real bottleneck?

Positioning + bench design. To get an honest production click rate:

1. The production bench needs a pre-flight check that its hardcoded coords actually point at icons on the current screen. Either: navigate to page 1 explicitly, OR template-match the current home page and pick targets accordingly.
2. Positioning accuracy (the residual ≥ 90 px on NO_LAUNCH trials) is the remaining variable. The trajectory data already collected for Stage 1 could in principle feed a *correctly-integrated* learned ballistics — but the cold-start integration that failed in 1.6 needs the redesign described in `2026-05-31-pointer-accel-1.6-fails.md`.

## Next-step recommendation

- **3.2 / 3.3 / 3.4 are unneeded** as currently scoped — there's no tap-registration brittleness to investigate.
- Promote a single Stage-3.5 step: **fix the production bench to verify its target coords are on real icons.** Re-run baseline against that. Whatever number falls out is the actual today click rate. From there, prioritise either positioning improvements (back to learned-ballistics redesign) or Stage-2 v12.1 absent-cursor work.
