# v12 76 px val plateau — the model is fine, the eval is broken

The plateau is a measurement artifact. Live, the saved `ml/cursor-v12.pt` has a
**median val error of 4 px in INPUT (768×480) space ≈ 10 px native (1920×1080)**,
i.e. better than v11's 13 px on human-labelled data. The "76 px" comes from a
silent MPS-vs-CPU device-mismatch bug in the trainer's val loop.

## H1 — label clustering: not the cause

`synth-train` (n=10000) and `synth-val` (n=5000) have nearly identical cursor-position
distributions: x ∈ [616,1303] mean≈964 std≈171, y ∈ [56,1023] mean≈554 std≈224.
There are 9766/4945 unique (x,y) pairs — minimal duplication. If the model
collapsed to the train mean, median val dist would be 108.6 px (INPUT). Measured
is 4 px, so this isn't it. (The x-range constraint comes from synth frames having
a centered portrait scene; this is real and matches val, not a bug.)

## H2 — centroid-vs-tip offset: cannot explain 76 px

Labels visually align with the cursor sprite (`/tmp/synth_val_0[1-6]_label.jpg`).
Any systematic offset would be ≤ the known 13 px centroid bias, not 76 px.

## H3 — regression to image-center bias: REJECTED

Ran `ml/cursor-v12.pt` on 5000 val rows via standalone PIL inference.
pred_x std=171.2, pred_y std=224.4 — matches the label distribution almost
exactly. Per-frame predictions track labels: first 10 pred-vs-label pairs are
within ~3 px each. The heatmap head is doing its job.

## H4 — train/val distribution mismatch: small, not the cause

7 older sources contribute ~1000 rows vs 10k synthetic, so train is
~90% synth. Val is 100% synth — well-aligned.

## H5 — coordinate transform bug: not it

`W_orig, H_orig = img.size` returns 1920×1080 from `Image.open(...).convert("RGB")`
for every synth frame inspected. Labels round-trip correctly into heatmap space.

## Root cause

**Trainer val loop subtracts MPS tensors from CPU tensors and PyTorch 2.12 on
Apple silicon silently returns garbage** (the CPU operand is treated as 0).
Lines 319–336 of `ml/train-cursor-v12.py`: `pred_x, pred_y` come back from
`decode_heatmap_peak` on MPS; `tgt_x_hm = torch.zeros(img.size(0))` is built on
CPU. The expression `(pred_x[i] - tgt_x_hm[i])**2 + (pred_y[i] - tgt_y_hm[i])**2`
mixes devices.

Reproduced in isolation:
```python
a = torch.tensor([0., 10., 20.], device="mps"); b = torch.tensor([0., 10., 20.])
(a[1] - b[1]).item()   # → 10.0   (should be 0.0)
```
The reported 76 px values are essentially `4 * sqrt(target_hm_x**2 + target_hm_y**2)`
of the LABEL alone — i.e. the distance from heatmap origin (0,0) to the target,
which by construction matches the cursor's std-dev pattern. That's why the
"plateau" appears stable: it's a deterministic function of label positions, not
of model quality.

When the distance is computed correctly (using `.item()` first, or moving tgt to
DEVICE), median goes from 62 → 4 px on the same batch.

## Fix recommendation

1. **Do not retrain.** `ml/cursor-v12.pt` is already at ~10 px native median —
   on par or better than v11.
2. **Fix the val loop** at `ml/train-cursor-v12.py:319-336`. Two minimal options:
   - Move targets to the model device: `tgt_x_hm = torch.zeros(img.size(0), device=DEVICE)` (and same for `tgt_y_hm`), OR
   - Compute on Python floats: `d = ((pred_x[i].item() - tgt_x_hm[i].item())**2 + (pred_y[i].item() - tgt_y_hm[i].item())**2) ** 0.5`.
3. **Add a guard test** that asserts `pred_x.device == tgt_x_hm.device` before
   the distance reduction (and replicate for any future trainer). Worth porting
   to v11's evaluator too — anything that ran on MPS post-PyTorch-2.12 may have
   reported inflated distances.
